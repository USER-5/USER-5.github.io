What’s the Point of Comments?

I mean this question geniuinely - where do code comments fit in these days?

If you think about it, we’ve had comments as programmers since very early on. And back in the day, they were crucial. We were often limited in how many characters we could make our variable and function names, so we had to explain our code through comments.

Today, we have the freedom to name our variables (almost) anything we like, so we can end up with nuggets of brilliance such as userManagerConfigFactoryV2. Fantastic… Regardless, through naming our variables well, we can help clarify its purpose.

Comments considered harmful?

So we don’t need comments, but what’s the harm in including them? Well, sometimes comments can actually cause confusion. Here’s a hypothetical.

You leave a comment on a piece of code. Naturally, it’s an excellent comment which is beautifully written and elegantly explains why the code is doing what it’s doing. It’s probably even got a diagram because you’re such a rockstar.

Some months later, your coworker (or future self) is doing a large refactor. They’re using refactoring tools to perform this change, and verifying their work isn’t breaking anything thanks to your codebase’s tests (as they should).

Unknown to them, they’ve just subtly changed the code under your comment, and now the flow is slightly different. It produces identical results (thanks to the refactor), but your excellent comment is now talking nonsense.

Congratulations! Your comment is now out-of-sync with your code, and nobody noticed!

So I should Avoid Comments?

Let me first state the obvious here: the people reading your code (and your comments) are likely to be programmers. And as programmers, they’re very good at reading and understanding what code does - especially if it’s well structured. And code doesn’t lie.

Now, you’re probably expecting me to say something along the lines of

If you structure your code properly, and name your functions and variables well, you don’t need comments at all!

And… sure… That’s one way to go. And this seems to be the opinion of quite a few people. But it’s not very realistic.

You’re also almost certainly going to come across code that just cannot be explained without a comment. Especially if the code needs to be performant.

And, to re-use the one joke in our field; there are two hard things in computer science:

  1. Cache invalidation
  2. Naming things
  3. Off-by-one errors

With good tooling, we can avoid the first and last one. Unfortunately, naming things well may always be a problem for humans (though I’ve seen people attempt to tool their way around this with generative AI - not with much luck).

What About Documentation?

Documenting your code with special comments (docstrings, javadoc, jsdoc, etc.) is a great way of keeping your documentation in-sync with your code (for the most part). And if you need documentation, I think this is your best option.

One honourable mention in this space is the use of testable examples in your documentation. These can remind developers to update the documentation when they change the code - although ideally your interface doesn’t change that much.

Verdict

I’m afraid this is going to be one of those wishy-washy “we should all learn to get along” type answers. Comments aren’t really going anywhere, and they’re still necessary in a large number of cases. At the same time, they are often overused by newer developers, and under-used by more experienced developers.

Perhaps all I can offer is this advice: prefer explaining your code with the code, instead of a comment.
