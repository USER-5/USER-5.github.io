We Don’t Need Documentation

…we don’t need no source control

Couldn’t help myself - it’s a good earworm.

Let’s be honest; documentation is something that not many people like to write. Even I, as someone who is voluntarily writing about programming right now, don’t like writing documentation most of the time. I recently had a discussion about documentation - where and why it’s important - and thought I’d turn it into a full article.

Introduction

The holy text for software project managers in the 21st century is arguably the agile manifesto. And don’t get me wrong; it’s been a wonderful thing for coders and leaders alike.

But you only need to get to the 2nd value to find this:

Working software over comprehensive documentation

Now I think this is a good value, and it encourages you to write readable code. The values are also explicitly intended to be preferences, not rules. But there’s a tendancy to use this as an excuse to write little-to-no documentation.

Make yourself redundant

No, really.

You should work to make yourself redundant. If you’re a good developer, there will always be new challenges awaiting you.

This means leaving information for the poor souls who have to pick up on your past work. If you don’t - they will be contacting you endlessly, pulling you away from that shiny new project that you finally got to start.

Code never lies, but it can take a long time to get to the point

As a developer, we are used to reading code in order to understand how it works. And the beautiful thing about code is that it doesn’t lie about what it does.

Unfortunately, that doesn’t mean that it’s very efficient at telling you what it’s doing at a larger scale. And let’s face it - not everyone can write readable code in the first place.

Documentation provides us a way of explaining code that frees us from the codebase itself. We can use diagrams to map out systems, and stories and analogies to reason about decisions.

Documentation should deliver value

Some people will say you should document everything, and that’s one option. Perhaps we’ll keep that idea just over here next to the bin in case we need it later…

In my opinion, your guiding star should be document things that will be useful later. Which seems obvious, but can often help weed out unnecessary wiki pages and busywork.

In addition to this - your entire process for documentation should focus on delivering value. If what you’re writing will never be read - then there’s no point. If your documentation templates are so dense that no-one ever creates a page - then you’re shooting yourself in the foot. If your wiki is guarded by reviewers who pick up on every typo - then… you get the idea.

Explaining a picture is worth like a thousand pictures

There’s few things more disheartening than someone droping a link to a wiki article and saying It’s in here - have a read.

Your documentation should stand on its own, yes. But it’s way easier to learn when you have the author helping explain it to you. This is also a good opportunity to find gaps in the explanation.
